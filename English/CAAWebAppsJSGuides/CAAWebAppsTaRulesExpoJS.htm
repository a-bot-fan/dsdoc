Authorized API Identification, Usage, Deprecation, and Stability
Technical Article

Abstract

This article gives you rules regarding Authorized JavaScript API identification and usage.

Authorized APIs

Dassault systèmes provides JavaScript APIs that can be used to create web applications. The list is available in the JavaScript API Reference. Only the APIs available in the Encyclopedia documentation can be used.

Authorized APIs Deprecation

When an Authorized API becomes obsolete, it is deprecated. Although this should be exceptional, the number of the Authorized API makes it happen more frequently than wished. The deprecation process runs as follows:

  • A deprecated API can be any Authorized API (module, class, function, method, typedef, data, and so on)
  • An Authorized API is deprecated starting from the General Availability (GA) of a new release, and never starting from a Fix Pack (FP)
  • A deprecated API remains available during a two-release life time starting from the one for which it is deprecated.

    In 3DEXPERIENCE a release can be identified by its marketing name, eg: 3DEXPERIENCE R2016x, or its code level, eg: R418. Correspondence between marketing names and code levels is described here. In the following we will use the code level that offers an easier understanding of the sequence order of the releases. For example, an Authorized API deprecated starting from R418 code level will remain available without modification during the whole R418 and R419 life times.

  • A deprecated API shows its replacing Authorized API that web applications must use instead.

    This replacing Authorized API is of course available at the deprecation time. The two-release life time is dedicated to developers to modify their web applications using the replacing Authorized API. Deprecated APIs are available from the Encyclopedia Reference section that includes a deprecated index for each API language.

  • A deprecated API issues a warning when web application are running.
  • A deprecated API is removed from the Authorized APIs two releases after the one it was deprecated. For example, an API deprecated starting from R418 GA will be removed on R420 GA

Authorized APIs Stability

Web application compatibility depends on Authorized API stability between FPs and releases.

Authorized APIs are checked in a such a way that:

  • They are run time upward compatible between the Fix Packs (FPs) of a given release.
  • They are run time upward compatible between two subsequent releases, as far as the deprecation process doesn't take place.

The possible modifications between two successive FPs:

  • Add new module, new class, new method, new data,
  • Add NON mandatory new option as input of a method,

The possible modifications between two successive releases:

  • Same as above PLUS:
  • Declare modules / classes / methods as deprecated provided that the function is replaced with another one.
  • Remove or de-expose deprecated APIs after two releases.
